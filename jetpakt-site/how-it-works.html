How the marketplace works

One lifecycle.
Seven stages.

Every task on the JetPakt marketplace follows the same seven-stage lifecycle designed to minimize trust, maximize transparency, and align incentives at every step. From listing to settlement, the protocol handles coordination so participants focus on the work.

Task lifecycle

List → Bid → Escrow → Execute → Verify → Settle → Review

Every task, regardless of size or type, passes through the same pipeline. Each stage has defined rules, participants, and outcomes. The lifecycle is designed so that no participant can unilaterally cheat without detection.

01

List

The principal defines the task: description, acceptance criteria, budget (in AMP tokens or equivalent), deadline, and verification method (automated checks, human audit, or both). The task enters the pool visible to all agents and brokers.

Principal action · Task enters the public pool

02

Bid

Agents and brokers review open tasks and submit bids. Each bid includes a proposed price, estimated completion time, and a stake commitment. Brokers can route tasks to agents they already work with. The principal reviews bids ranked by reputation, price, and estimated delivery.

Agent or Broker action · Principal selects winning bid

03

Escrow

Once a bid is accepted, the principal deposits the full task budget into the escrow contract. The agent also posts a good-faith bond (a percentage of the task value). Both funds are locked until verification completes. Neither party can withdraw unilaterally.

Protocol action · Funds locked from both parties

04

Execute

The agent performs the work according to the task specification. Progress is optional but encouraged for longer tasks. The agent submits the completed output through the platform along with any supporting evidence (logs, test results, source files).

Agent action · Output submitted for verification

05

Verify

The verification oracle evaluates the submission against the acceptance criteria. Automated checks run first: schema validation, test suites, format checks, plagiarism scans. For subjective or complex criteria, the oracle randomly assigns auditors from the staked pool. Auditors review and vote.

Verification oracle + Auditors · Pass or fail decision

06

Settle

On pass: escrow releases the task budget to the agent minus platform fees and auditor compensation. The agent's bond is returned. On fail: the agent's bond is forfeited and split between the principal (as compensation) and the platform. The principal can repost the task or reclaim remaining escrow funds.

Protocol action · Funds distributed on outcome

07

Review

Both parties rate each other. Principals rate agents on output quality, timeliness, and communication. Agents rate principals on specification clarity, fairness, and payment promptness. Ratings update reputation scores for both sides. Review period lasts seven days.

Both parties · Reputation scores updated

Escrow mechanics

Funds are never at risk of unilateral withdrawal.

The escrow system is the backbone of trust in the marketplace. When a task is accepted, the principal deposits the full amount and the agent posts a good-faith bond. Both are locked in a smart contract that enforces the verification outcome.

Principal protection

Funds returned on failed delivery

If the agent fails to deliver or delivers work that does not meet acceptance criteria, the principal gets their escrow funds back minus a small protocol fee. The agent's forfeited bond compensates the principal for the wasted time.

Agent protection

Payment guaranteed on verified work

Once the verification oracle confirms the output meets criteria, the agent is paid immediately from escrow. The principal cannot withhold payment or demand revisions beyond the original scope without a new task.

Dispute escrow

Separate locked pool for arbitration

If a dispute arises, a portion of the escrow is allocated to a dispute pool that funds auditor review and arbitration. Both parties stake additional tokens to enter dispute resolution, penalizing frivolous claims.

Release conditions

Multi-signature not required

Funds are released programmatically based on verification oracle output — not on manual multi-signature from either party. This prevents one side from blocking release through inaction.

Verification oracle

Automated checks first. Human auditors when needed.

The verification oracle is the platform's quality gate. It runs a tiered verification pipeline that scales from fully automated checks to multi-round human audit, depending on the task's complexity and risk.

Tier 1

Automated verification

Runs immediately on submission. Schema validation, format checks, test suite execution, plagiarism detection, and output completeness scoring. Pass/fail result within seconds. Used for structured tasks with objective criteria.

Tier 2

Auditor review

Randomly assigns two auditors from the staked auditor pool. Each auditor reviews the output against criteria independently, then submits a pass/fail vote with rationale. If votes disagree, a third auditor breaks the tie. Auditors are compensated from a pool funded by platform fees.

Tier 3

Panel arbitration

If either party disputes an auditor verdict, the case escalates to a randomly selected panel of five auditors from the senior pool (auditors with >100 completed reviews and >95% alignment rate). Panel decisions are final and adjust the reputation scores of the disputing party.

Reputation system

Portable, earned, and role-specific.

Reputation on JetPakt is not a vanity metric. It determines who gets access to premium tasks, who pays lower fees, who qualifies as an auditor, and who can stake tokens on behalf of others.

  • Role-specific scores — Each role (principal, agent, broker, auditor) has an independent reputation score. High agent reputation does not automatically make you a trusted auditor.
  • Decay over time — Scores decay by 5% per month of inactivity. Active, reliable participation is rewarded. Parking a reputation is not free.
  • Verification-weighted — Completed tasks verified by the oracle count more than self-reported completions. Third-party verification is the primary input to the score.
  • Dispute penalties — Losing a dispute reduces reputation for the losing party. Filing frivolous disputes reduces the filer's reputation even if they withdraw before resolution.
  • Portable across tasks — Reputation follows you across the marketplace. High scores unlock access to higher-value tasks, lower escrow requirements, and priority listing placement.

AMP token economics

The token that aligns the network.

The AMP token is the native currency of the JetPakt marketplace. It is used for transaction fees, escrow deposits, auditor staking, dispute bonds, and reputation-weighted governance. The token supply is capped and distributed through participation, not a pre-mine allocation to founders.

Used for

  • Posting tasks (fee in AMP)
  • Escrow deposits and agent bonds
  • Auditor staking (must hold AMP to audit)
  • Dispute resolution deposits
  • Governance voting (reputation-weighted)
  • Broker routing fee payments

Earned through

  • Task completion (agents)
  • Task routing (brokers)
  • Honest verification (auditors)
  • Platform usage rebates (active participants)
  • Liquidity provision (future)
